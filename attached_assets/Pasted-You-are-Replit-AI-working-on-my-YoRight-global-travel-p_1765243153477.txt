General mindset

– Avoid quick hacks. Prefer clean, explicit, well-documented solutions.

	1.	Overall architecture and modularity
– Refactor the codebase into clear high-level domains/modules, for example:
• core/common (shared types, utilities, error classes, result wrappers)

• catalog (destinations, properties, room types, static content about accommodations and cities)
• pricing & margin (markup rules, commissions, discounts, currency conversions)

	2.	Supplier and provider abstraction layer
– Build a dedicated “supplier adapter” layer that hides the details of external providers. Think in terms of interfaces like AccommodationSupplier, PricingSupplier, ContentSupplier.
– Implement a provider registry/factory that can select the correct supplier based on configuration, region, or A/B test group.
– Make sure the rest of the system never directly calls a single provider SDK; it must go through these interfaces.
– Prepare the code so we can plug in additional accommodation providers later (e.g. other wholesalers or direct contracts) without breaking existing flows.
– Support merging and normalizing data from multiple suppliers into a unified internal model (unified hotel ID, normalized amenities, room types, board types, etc.), even if initially only one provider is active.
	3.	Pricing, margin, and currency engine
– Create a robust pricing engine that:
• Works with multiple currencies and can use a daily exchange rate source (abstracted, so implementation can be swapped later).
• Supports global default markup, region specific markup, partner specific markup, and property specific overrides.
• Can define rules like: minimum margin per booking, maximum percentage increase, rounding rules, and thresholds for displaying “special deal” labels.
• Can support promo codes, coupons, and partner-specific discounts in the future.
– Make the pricing engine deterministic and fully testable: given an input (base net rate, currency, rule set) it must output the final price, a breakdown, and explanations (for debugging and admin UI).
– Implement a clear data model for margins and rules, including effective date ranges (so we can schedule future price campaigns).
	4.	Reliability, resilience, and failure handling
– Implement robust error handling and resilience patterns for all external calls:
• Use timeouts, retries with exponential backoff where appropriate.
• Implement circuit breakers for providers so that if one provider starts failing heavily, we stop hitting it aggressively and use fallbacks or show graceful degradation to the user.
• Add standardized error classes (e.g. ExternalServiceError, ValidationError, DomainError, PermissionDeniedError) and a central error handling middleware for the backend APIs.
– For every critical operation (availability check, booking creation, cancellation), define clear stages and make operations idempotent where possible, to avoid duplicate bookings if the client retries.
– Use correlation IDs (request IDs) injected at the edge of the API and propagated through logs and external calls so we can trace a single user action end-to-end.
– Ensure the system always fails in a user-friendly way: clear messages, no raw stack traces, and accurate instructions on what the user can do.
	5.	Performance and caching strategy
– Design and implement a layered caching strategy:
• In-memory cache on each instance for ultra-fast repeated reads.
• External cache (e.g. Redis or equivalent) for cross-instance caching of expensive queries and external API responses.
• Well-defined cache keys and TTLs for search results, static catalogs (destinations, cities), exchange rates, and frequently used metadata.
– Implement cache invalidation strategies for each resource type. For example:
• Short TTL for highly volatile data (availability and prices).
• Longer TTL for static content (city descriptions, static amenities).
– On the frontend, implement techniques like:
• Code splitting and route-based lazy loading.
• Prefetching of data for likely next steps (for example, prefetch details for hotels that are visible in the viewport on the search results page).
• Debounced search inputs to avoid spamming the backend.
– Add basic performance budgets and metrics: measure and optimize for time-to-interactive, key API latencies, and bundle sizes.
	6.	Security, privacy, and access control
– Harden the entire stack against common web vulnerabilities:
• Input validation and sanitization on all API endpoints and query parameters.
• Avoid SQL injection with parameterized queries or ORM safeguards.
• Apply appropriate content security policy (CSP), XSS protection, and safe handling of user-generated content.
• Secure cookie configuration and proper session handling if sessions are used.
– Implement fine-grained role-based access control (RBAC):
• Distinguish between regular users, support agents, admins, finance team, and potential B2B partners.
• Enforce “least privilege” so each role only has access to what they need (e.g. support can view bookings but not modify system-wide margin rules).
– Build a clear audit log system:
• Track who changed what, and when, for critical entities (pricing rules, booking status changes, cancellations, refunds, configuration).
• Store audit logs in an append-only structure with timestamps and actor metadata.
– Prepare the system for data protection and privacy requirements (e.g. GDPR-style):
• Implement mechanisms to export a user’s data in a structured format.
• Implement mechanisms to anonymize or delete a user’s personal data while keeping necessary accounting information pseudonymized.
– Establish a secure configuration pattern: environment variables for secrets, no secrets in source code, and clear documentation of all required secrets.
	7.	Observability: logging, metrics, tracing
– Implement structured, machine-readable logging across backend services:
• Use a consistent log format with fields like timestamp, level, message, requestId, userId (if known), route, and providerId where applicable.
• Separate logs by concern (application, access, audit, error).
– Expose application metrics suitable for a metrics system (request counts, latencies, error rates, etc.).
– Integrate basic tracing concepts (even if only internally for now) so that we can understand the path of complex operations (search → multiple provider calls → merge → response).
– Add critical business metrics counters: number of searches, number of booking attempts, success rate by provider, total margin over time, and top destinations by search and booking.
	8.	Advanced frontend experience and quality
